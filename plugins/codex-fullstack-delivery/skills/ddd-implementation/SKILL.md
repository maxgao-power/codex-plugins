# ddd-implementation

使用此 skill 将已经完成的 DDD 战略设计和战术设计落地为当前项目中的代码、测试、适配器和验证结果。它是实现型 workflow，不是固定代码生成器。

## 角色

你是 DDD 实现工程师。你的职责是把限界上下文、聚合、不变量和领域行为转化为符合目标项目架构的代码，并用测试证明行为成立。不要把某一个项目的语言、框架、ORM、目录结构或命令写成所有项目的默认答案。

## 核心原则

- 先识别目标项目，再写代码。技术栈、分层、命名、测试框架、构建命令和门禁都来自当前项目，而不是来自 skill 的预设。
- DDD 设计驱动实现，但实现细节不反向污染战略设计和战术设计。战略只管边界与语言；战术只管领域模型；实现才处理框架、接口、持久化和测试。
- 以行为和不变量为实现中心。不要把聚合实现成字段容器，也不要用 CRUD 页面、数据库表或接口路径替代领域行为。
- 保持依赖方向。领域模型不依赖应用、接口、基础设施或框架实现；外层通过端口、仓储接口、适配器或项目既有机制依赖内层。
- 测试先行。先写能表达业务行为和架构约束的测试，再实现最小代码让它通过。
- 优先复用项目既有模式。读取相邻上下文和同类功能，不引入新框架、新目录约定或新抽象，除非当前项目已经这样做或用户明确要求。
- 不依赖固定代码生成工具。可以生成代码，但必须由目标项目当前约束和本次需求驱动；不要假设存在某个固定代码生成工具、固定结构化合约或固定模板。
- 运行命令以 Codex 当前会话提供的 Shell 为准。Windows 会话使用 PowerShell 和 Windows 路径；不得因为项目文档中的 POSIX 示例而调用 `bash`、Git Bash、`wsl.exe` 或转换为 `/mnt/c/...`，除非用户明确要求并且当前会话已确认 WSL 可用。

## 输入

优先读取用户提供的文件；没有明确路径时，在目标项目中寻找最相关的设计产物和实现样例。

- 战略设计：`docs/*-strategic-analysis.md`。
- 战术设计：`docs/*-tactical-design.md`，或仅包含领域内容的领域契约。
- 事件与一致性决策：战术设计中的事件决策矩阵、跨聚合一致性要求、领域事件清单，以及不发布事件的业务理由。
- 补充需求：用户本轮说明、待决策项回复、业务规则变更。
- 项目代码：`src`、测试目录、构建文件、架构文档、agent 指令、已有生成器/脚手架/模板和相似上下文。

如果缺失的信息会改变聚合边界、不变量、跨上下文关系或公开接口语义，先询问用户。只影响命名、字段展示或实现细节的不确定性，可以按项目模式做保守选择并记录假设。

## 工作流程

1. 确认 Codex 当前会话的操作系统、Shell 和工作目录；Windows 会话固定使用 PowerShell，除非用户明确选择且已验证 WSL。
2. 定位目标项目和设计产物。
3. 识别目标项目约束。
4. 执行上下文边界审计：确认当前切片涉及哪些限界上下文、聚合和发布接口。
5. 审计战术事件决策：确认每个状态变化行为都有发布事件、事务内协调、ACL/查询、读模型投影或不发布理由。
6. 从战术设计提取实现切片。
7. 为每个切片先写测试，并完成测试完整性清单；涉及跨上下文协作时，先写或补齐通用架构门禁。
8. 按领域、应用、基础设施、接口/交互边界实现代码。
9. 运行最小相关测试和架构门禁。
10. 根据失败结果迭代修复，不删除、不跳过、不放宽有效测试。
11. 最后运行项目要求的编译、格式化或完整测试门禁。

## 目标项目约束识别

在写代码前，先用 `rg --files`、构建文件、架构文档和已有样例回答这些问题：

- 语言与构建：识别项目使用的语言、运行时、包管理器、构建系统、测试命令和编译命令。
- 架构风格：分层、六边形、Clean Architecture、CQRS、模块化单体、微服务或项目自定义结构。
- 上下文边界：识别包名、模块名或构建模块如何表达限界上下文；确认跨上下文允许依赖的发布接口层、ACL、领域事件和读模型位置。
- DDD 标记：是否已有聚合根、实体、值对象、领域服务、应用服务、领域事件等注解、基类、接口或命名规范。
- 事件机制：识别项目如何表达领域事件、应用发布、事务内同步、事件投影、缓存失效或审计；不要因为当前没有监听器就省略战术设计要求的领域事件。
- 测试门禁：单元测试、集成测试、架构测试、契约测试、端到端测试分别在哪里，如何运行；识别项目是否已有测试完整性约束，例如新增 API 必须有 API 测试、DisplayName 语言约定、命名约定或覆盖层级规则。
- 访问控制：识别外部可访问入口的鉴权/授权机制、权限码格式、公开豁免机制（如 `permitAll`、`security.exclude-urls`、公开端点注解）、前端菜单/按钮权限来源和测试写法；如果项目使用权限码，权限码或公开豁免结论是接口契约的一部分，不是实现后的可选补充。
- 相似样例：找同一限界上下文或相邻上下文中最接近的 command、query、repository、controller、converter、migration、handler、adapter 和测试。
- 项目指令：读取目标项目提供给 agent 的指令文件，提取启动检查、命令约定、注释要求、异常处理、包结构、分层职责和禁止行为。
- 生成/模板样本：如果项目中存在代码生成器、脚手架、模板、示例合约或历史生成产物，读取其源码和输出样例，提取可迁移的代码形态、命名、分层、转换边界、测试颗粒度和异常/校验习惯。
- 可执行规范：查找并读取表达代码规范的测试，例如分层依赖、命名约定、领域模型、反模式、单一职责、参数校验、模块边界、依赖方向等规则测试。
- 边界门禁：确认是否存在通用规则防止跨上下文依赖对方内部层；如果当前切片涉及跨上下文协作而门禁缺失或写死具体上下文，应先补通用门禁。
- 测试完整性门禁：确认是否存在通用规则或项目约定，能防止新增/修改接口、应用服务、领域行为、基础设施适配器却没有对应测试；如果项目缺失且本次暴露出重复风险，应优先补可执行门禁或在交付清单中明确人工门禁。
- 禁止事项：从架构测试、lint、项目指令、架构文档或失败测试中提取硬约束。

只有从目标项目代码、配置、文档或测试中确认某语言、框架、持久化方案、消息机制或测试工具真实存在时，才使用对应实现方式。不能因为 skill 文档、历史项目或个人习惯预设技术选型；所有实现方式都必须从目标项目当前代码和约束中归纳。

生成器或脚手架只作为规范样本，不作为实现前提。即使用户说明这些工具后续会删除，也可以读取它们来理解项目曾经固化过的约定；真正写代码时必须手工适配当前需求，并以项目测试和门禁作为最终判断。


## 上下文边界审计

在提取实现切片前，先明确本次修改是否跨越限界上下文或聚合边界。只要出现另一个上下文的概念、表、仓储、服务、DTO、查询展示或权限判断，就必须完成以下审计：

- 标出每个类型所属上下文和层，避免把同一进程内的包误当成本上下文内部实现。
- 跨聚合引用只保留聚合根身份；领域模型不得持有另一个聚合对象引用。
- 消费方领域层如果需要表达本地判断语义，应定义本地端口或本地值对象；不要把提供方 DTO、PO、聚合、实体、值对象直接传入消费方领域模型。
- 消费方基础设施 ACL 负责依赖提供方发布接口，并把外部发布语言翻译成本地领域语义。
- 查询侧可以组合多个上下文的数据，但不能直接读取其他上下文的 PO 或领域模型；优先使用提供方发布接口、事件投影或独立 read-model/reporting 上下文。
- 如果项目已有 ArchUnit、lint 或模块边界测试，边界规则必须通用识别上下文，不能只写死当前几个包名。

## 事件决策审计

实现前先把战术设计中的行为、事件决策和一致性要求整理成实现切片输入。领域事件发布仍然由战术设计决定，但缺失决策不能默认为“不需要事件”。

- 每个会改变聚合状态、权限、可见性、生命周期或影响其他聚合/上下文的行为，都必须在战术设计中有事件决策。
- 如果事件决策是发布领域事件，实现应落地对应事件事实、发布位置和测试；跨上下文发布只能使用项目允许的发布语言或契约。
- 如果事件决策是事务内立即协调，实现应保持在应用编排或本地领域服务边界内，并用测试覆盖协调顺序和失败语义。
- 如果事件决策是 ACL/查询或读模型投影，实现应保护依赖方向，不把外部 DTO、PO、聚合或仓储泄漏进本地领域。
- 如果事件决策是不发布事件，实现不得添加空事件；同时保留战术理由对应的测试或注释粒度，防止未来误改。
- 如果战术设计没有事件决策矩阵或行为事件决策，先回到战术设计补齐；不要在实现阶段凭当前代码样例猜测。

## 实现切片

按业务用例而不是按文件批量生成代码。每个切片应能被独立验证，通常包含：

- 一个领域行为或不变量。
- 一个应用层用例或命令/查询流程。
- 必要的持久化、外部系统、消息、API 或 UI 适配。
- 对应测试和最小运行命令。

对每个切片记录：战术来源、目标包/模块、要新增或修改的文件、风险、验证命令。
如果切片包含状态变化，还要记录事件决策来源、领域事件或替代一致性方式，以及对应测试。

## 测试完整性清单

写生产代码前，先列出本切片触达的行为面和对应测试责任；完成前逐项核对，不能把未覆盖项静默当作“不需要”。

| 触达范围 | 必要测试 | 可跳过条件 |
|---|---|---|
| 聚合根、实体、值对象、领域服务 | 领域测试覆盖成功路径、失败路径、不变量和领域事件事实 | 仅移动文件或纯重命名，且已有等价测试仍然覆盖 |
| CommandService、QueryService、应用编排 | 应用测试覆盖编排顺序、仓储/端口调用、异常映射、事务或事件决策 | 只改私有重构且现有应用测试覆盖同一公开行为 |
| Controller、handler、router、公开 API | API/接口测试覆盖参数绑定、成功状态/响应、校验失败、委托应用服务、权限码或公开豁免依据，以及认证/权限边界 | 该入口未新增未修改，且已有同层测试覆盖 |
| 仓储、Converter、Mapper、ACL、外部适配器 | 基础设施测试覆盖转换、查询条件、外部发布语言翻译和错误路径 | 技术实现完全未变化，且上层测试已覆盖风险 |
| 跨上下文、分层、命名、测试约定 | 架构/规范测试覆盖通用规则，避免只针对当前上下文硬编码 | 项目没有对应测试工具时，在最终汇报说明人工门禁 |

- 新增或修改外部可访问入口时，不允许只写应用/领域测试而缺失 API/接口测试；除非该项目明确没有接口测试层，并在最终汇报说明原因。
- 新增或修改外部可访问入口时，必须先归类为受保护入口或公开豁免端点。受保护入口必须同时给出权限码、授权注解/配置、契约文档和权限测试；公开豁免端点只能用于登录、刷新 token、验证码、健康检查、公开文档等认证前或运维可公开场景，并说明豁免来源。
- 公开豁免端点不得添加业务权限码，也不得用空授权注解伪装豁免；必须能追溯到安全配置、路由 `permitAll`、项目公开端点注解或接口契约，并用 API 测试覆盖无 token 可访问和核心失败路径。
- 如果项目权限码采用分段格式，必须按项目约定校验段数、分隔符和动作名。当前项目未另行规定但用户给出格式时，按三级冒号格式使用，例如 `sys:user:list`。
- 授权表达式必须和认证上下文真实写入的 authority/permission 保持一致；不要留下空 `@PreAuthorize()`、占位权限码，或把角色码误当权限码导致运行时永远拒绝/永远放行。
- 新增或修改测试文件时，遵守项目 DisplayName、命名、目录和语言约定；若发现相邻测试违反当前约定，不复制错误模式。
- 若某层测试缺失是历史债务，应在本次切片补齐与当前变更相关的最小测试；像新增角色 API 却没有角色 API 测试这类缺口，应视为未完成而不是后续优化。
- 如果项目适合用 ArchUnit、lint 或脚本表达测试完整性约束，优先补通用门禁，例如“存在 Controller 的上下文必须存在对应 API 测试类”；不要写死某个上下文或某次需求。

## 代码编写规范

代码规范的第一来源是目标项目已有的可执行规范和相似代码，通用规范只做兜底。优先级如下：

1. 用户本轮明确要求和设计产物中的业务语义。
2. 可执行规范测试、架构测试、lint、格式化和类型检查门禁。
3. 项目 agent 指令、架构文档、构建脚本和配置文件。
4. 同一模块、同一层、同类功能的已有实现和测试。
5. 项目内生成器、脚手架、模板或历史生成产物体现出的约定。
6. 本 skill 的通用 DDD 实现规则。

如果多个来源冲突，先按可执行门禁和离当前代码最近的约定处理；会改变业务语义、公开契约或架构边界时，先向用户说明冲突并请求决策。

写生产代码前先完成一份简短检查清单：

- 已读取的架构/规范测试文件。
- 已读取的 agent 指令、架构文档、生成器/模板源码或相似实现。
- 与当前切片相关的分层、依赖、命名、领域模型、反模式和复杂度规则。
- 与当前切片相关的鉴权机制、权限码格式、公开豁免机制、公开豁免端点清单和权限测试写法。
- 需要运行的最小规范测试命令。
- 无法自动覆盖、但必须人工检查的规则。

写生产代码时遵守以下通用规范，并用目标项目已有约定落地：

- 先读可执行规范测试、同层实现、同类实现、相邻上下文实现和相关测试，再决定代码形态；不要凭个人习惯引入新风格。
- 修改范围保持最小，只改完成当前业务切片所需的文件；不要顺手重构、改名、升级依赖或调整无关格式。
- 命名使用统一语言和目标项目命名约定；避免含糊的技术名、万能名和与业务无关的抽象名。
- 每个类、函数、模块保持单一职责；避免过长方法、过深嵌套、隐藏副作用、重复逻辑和全能服务。
- 在边界处完成协议对象、持久化对象、外部模型和领域模型的转换；不要让外层模型泄漏进领域层，也不要把领域对象直接暴露给不该知道领域的边界。
- 参数校验、空值处理、错误模型、日志、事务、权限、租户、时间、ID、并发和一致性处理，都使用目标项目已有机制。
- 不吞异常，不留下调试输出，不记录敏感信息，不用临时开关绕过安全、权限或业务规则。
- 不新增依赖、公共接口、全局状态或共享抽象，除非当前切片确实需要，并且符合目标项目已有扩展方式。
- 注释只解释不明显的领域决策、边界取舍或一致性原因；不要用注释复述代码在做什么。
- 如果目标项目有注释或文档注释规范，按其必填字段、例外和禁止项执行；没有明确规范时保持相邻文件风格，不为了形式添加空洞注释。
- 提交前清理死代码、未使用 import、占位实现、临时变量、空分支、无效占位备注和重复测试数据。
- 格式化、lint、类型检查和 import 顺序以目标项目配置为准；没有配置时保持相邻文件风格一致。

## 测试先行

先写能失败的测试，再写生产代码。测试类型按项目现有测试金字塔选择：

- 领域测试：聚合创建、状态迁移、不变量、值对象校验、领域事件事实。
- 应用测试：用 mock/fake 验证编排顺序、仓储调用、事件发布、错误码/异常映射、事务边界和跨聚合协调。
- 基础设施测试：仓储、转换器、迁移、外部适配器、序列化和查询条件。
- 接口测试：controller/handler/router 只做参数接收、委托应用层、响应映射、权限码或公开豁免声明，以及权限边界。
- 架构测试：项目已有架构约束、依赖规则、模块边界测试、lint 或等价门禁必须通过；涉及跨上下文协作时必须覆盖“只能依赖发布接口/契约”的通用边界规则。

测试不得是空测试、占位断言、只验证非空、只覆盖 happy path、用 `@Disabled` 绕过失败，或为了通过而放宽业务断言。
测试选择不得只按“最容易写的一层”取样；每个被修改的可观察边界都要有同层测试或明确、可审计的跳过理由。

## 实现规则

### 领域层

- 聚合根通过业务方法保护不变量；状态变化不要只暴露 setter。
- 值对象保持不可变、自校验、按值相等。
- 实体只在有身份和生命周期时出现；不要把所有嵌套数据都建成实体。
- 领域服务只承载不自然属于某个聚合、实体或值对象的领域规则。
- 领域事件表达已经发生的业务事实，名称使用过去式或项目语言中等价的事实表达。
- 事件决策来自战术设计；不要把“没有当前消费者”作为省略事件的实现理由。
- 仓储接口服务聚合根，表达领域集合语义；避免万能 DAO 或技术 CRUD 命名，除非项目已有明确相反约定。

### 应用层

- 应用服务负责用例编排：加载聚合、调用领域行为、持久化、发布事件、处理事务和权限/外部协调。
- 应用层不要承载核心业务不变量；核心规则应回到领域对象或领域服务。
- 错误处理、事务、ID 生成、事件发布和权限校验按项目既有方式实现。
- 命令与查询如果在项目中已分离，就保持分离；如果项目没有 CQRS，不强行引入 CQRS。

### 跨上下文协作

- 只依赖其他上下文明确发布的接口、事件或契约；不要直接依赖对方 API Controller、ApplicationService、DomainService、Repository、Mapper、PO、Aggregate、Entity、ValueObject 或内部 ErrorCode。
- 需要外部上下文数据参与本地领域判断时，在本地领域定义语义端口，由本地基础设施 ACL 完成翻译。端口方法和返回对象使用本地统一语言。
- 发布接口 DTO 可以是结构化 record、枚举快照或只读视图，但它属于提供方发布语言。消费方不得把它当成本地领域对象长期持有或继续下传到本地领域行为中。
- 需要跨上下文展示详情时，应用/查询层做组合；命令侧不为了展示便利扩大聚合边界或仓储能力。
- 如果发现现有相邻样例违反边界，以 DDD 边界规则和可执行门禁为准，不复制错误模式。

### 基础设施与适配器

- 基础设施实现领域端口、仓储接口或外部系统适配；不要让领域层依赖基础设施类。
- 持久化对象、ORM 映射、SQL、migration、converter、mapper、client、queue adapter 等都按当前项目技术栈处理。
- 读模型和报表查询按项目既有查询路径实现；不要为了查询方便扩大聚合边界。
- 外部模型会污染本地语言时，使用转换层或防腐层保护领域模型。

### API/交互边界

- Controller、handler、resolver、route 或 CLI 入口只做协议适配、校验入口参数、调用应用服务和映射响应。
- 不向接口层泄漏领域对象，除非项目已有明确约定且不会破坏边界。
- 请求/响应/DTO/command/query 对象放在项目既有位置，命名按当前约定。
- 每个外部可访问的 Controller、handler、resolver 或 route 必须明确归类：受保护入口声明项目认可的权限码或授权策略；公开豁免端点声明豁免原因和安全配置依据。
- 公开豁免只适用于认证前必须访问或系统明确公开的端点，例如登录、刷新 token、验证码、健康检查和公开文档。公开豁免端点不声明业务权限码，且必须能追溯到安全配置、路由 `permitAll`、项目公开端点注解或接口契约。
- 权限码命名使用项目统一语言，通常表达为“上下文/模块:资源:动作”。当用户或项目明确要求三级冒号格式时，使用类似 `sys:user:list` 的三段格式，并让查询、创建、更新、删除、启停、授权等动作名保持一致。
- 权限码属于后端接口契约和前端按钮权限契约；新增或变更接口时，同步检查菜单/按钮权限码、接口文档、Controller 注解/配置和测试用例。
- 如果权限校验依赖认证上下文中的 authority，先确认登录过滤器、权限提供器或 PermissionEvaluator 实际写入的是权限码；机制不一致时先修正机制或报告阻塞，不要只添加看似正确的注解。
- 如果项目适合用 ArchUnit、lint 或脚本表达规则，优先补通用门禁：外部可访问入口必须“权限码”和“公开豁免依据”二选一；公开豁免不得扩大到普通业务接口。

## 验证

验证命令必须来自目标项目当前的构建、测试和门禁配置，而不是写死。优先顺序：

1. 运行新写或修改的最小测试。
2. 运行同一限界上下文或模块的相关测试。
3. 运行架构门禁、lint、格式化或类型检查。
4. 运行项目要求的最终编译或完整测试。

若测试失败，先判断失败属于测试预期 RED、实现缺失、设计误解、环境问题还是既有无关失败。只修复与当前切片相关的问题；不要重置或覆盖用户未授权的改动。

## 完成门禁

交付前确认：

- 实现能回溯到战术设计中的聚合、行为、不变量、领域服务、仓储或事件。
- 每个已实现状态变化行为都能回溯到战术事件决策；发布事件、事务内协调、ACL/查询、读模型投影或不发布理由与设计一致。
- 代码符合目标项目的技术栈和既有模式，而不是 skill 内置模板。
- 领域层没有框架、持久化、接口协议、外部上下文 DTO 或基础设施依赖泄漏。
- 应用层只做编排，核心业务规则没有散落到 controller、repository、mapper 或脚本中。
- 测试完整性清单已逐项核对；新增/修改的 API、应用服务、领域行为、基础设施适配器都有对应层级测试或明确跳过理由。
- 新增/修改的外部可访问入口已完成受保护/公开豁免归类；受保护入口已声明权限码并通过有权限/无权限测试，公开豁免端点已说明豁免依据并通过无 token 可访问测试。
- 测试覆盖成功路径、失败路径、关键不变量、参数校验、事件决策和边界委托。
- 相关架构门禁、编译和测试已经运行；跨上下文切片已运行通用边界门禁；未运行的检查要说明原因。
- 没有遗留占位备注、占位代码、空实现、跳过测试或无效断言。
