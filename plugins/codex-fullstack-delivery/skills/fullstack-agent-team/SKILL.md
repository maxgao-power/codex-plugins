# 全栈 Agent 团队

使用本 skill 时，Codex 扮演技术负责人，协调前端 Agent、后端 Agent 和测试 Agent 完成全栈功能开发、接口对接与自动化验收。默认目标是契约优先、并行推进、测试闭环、主上下文瘦身。后端涉及新业务能力、领域规则或公开接口语义变化时，先用插件内 `ddd` 流程编排完成领域设计和实现，再把接口契约交给前端对接。

## 开发前强制门禁

开始任何代码修改前，必须先识别并向用户提供前端和后端项目地址：

- 前端项目地址：本地路径或仓库 URL、分支、可发现的运行命令和测试命令。
- 后端项目地址：本地路径或仓库 URL、分支、可发现的运行命令和测试命令。

在这两个地址明确之前，不要修改前端或后端代码。如果用户只提供了一侧项目地址，先询问缺失的一侧。如果当前工作区或权限无法访问必要项目，明确说明阻塞并请求用户提供可访问路径、仓库或项目内容。

## 强制调度门禁

完成项目地址识别后、开始任何前端或后端实现前，技术负责人必须先完成调度门禁。不要把“后端 Agent”“前端 Agent”当作主会话内的角色扮演；它们必须是独立 sub-agent、后台线程、用户已打开的项目线程，或明确输出给用户转交的任务包。

调度顺序：

1. 先寻找可用调度能力：如 `multi_agent_v1.spawn_agent`、Codex 线程工具（`create_thread`、`fork_thread`、`send_message_to_thread`）或用户提供的现有项目线程。若工具尚未暴露，先用工具发现能力搜索 sub-agent、multi-agent、thread、delegate、parallel agent 相关工具。
2. 若 `multi_agent_v1.spawn_agent` 可用，优先创建独立 `worker`：后端 Agent 负责后端项目、DDD、接口契约和后端测试；前端 Agent 负责前端项目、页面/API client 和前端测试；实现完成后再创建或复用测试 Agent 做 smoke 与 E2E。
3. 若 Codex 线程工具可用且项目需要分线程执行，创建或驱动后端项目线程、前端项目线程和测试线程，并要求交接摘要回传给技术负责人。
4. 若没有可用 sub-agent 或线程工具，但用户已打开对应项目线程，输出可直接发送的任务包，并要求用户把交接结果贴回当前技术负责人线程。
5. 若既没有可用调度工具，也没有可用项目线程，必须停止在实现前，只输出后端、前端和测试任务包及阻塞原因；不得退化为主会话直接完成前端或后端实现。

主会话职责边界：

- 只做项目发现、任务拆分、接口契约来源确认、调度、交接审查、冲突裁决、集成决策和最终验收总结。
- 不直接修改前端或后端业务代码，不直接补齐测试实现，不在主上下文执行长时间实现工作。
- 只有用户明确要求主会话处理某个单侧、窄范围、非全栈协调任务时，才可跳过团队调度，并在回复中说明跳过原因。
- 每次启动或转交任务后，记录调度方式、agent id 或 thread id；如果只能输出任务包，也记录“任务包待用户转交”。

## 团队角色

除非用户明确只要求某个角色，否则默认由 Codex 担任技术负责人。

- 后端 Agent：负责 DDD 流程编排、API 设计、服务逻辑、数据模型变更、接口文档、后端测试和后端运行说明。开发新业务功能、领域规则、聚合行为或跨上下文协作时，必须先使用 `ddd` 编排 skill 推进战略设计、战术建模和实现落地；只有纯技术修复、配置调整或无领域语义的小改动才可跳过 DDD 流程，并在交接中说明原因。
- 前端 Agent：负责页面、组件、状态管理、API 调用、前端测试和用户侧集成体验。
- 测试 Agent：负责自动化测试补齐与执行，按后端接口自动化、前端 service 映射测试、接口联调 smoke 脚本、浏览器 E2E 顺序输出测试结果。
- 技术负责人：负责任务拆分、执行顺序、接口契约审查、冲突处理、集成验证和最终交付总结。

保持职责边界清晰。某个角色可以阅读另一侧代码来理解对接方式，但代码修改应尽量留在所属项目中。确实需要跨侧小改时，由技术负责人说明原因。

前端 Agent 和后端 Agent 在实现代码前必须读取所属项目的 `AGENTS.md`，并按其中的架构、命名、测试、提交和安全规范开发；如果项目没有 `AGENTS.md`，再读取 README、CLAUDE.md 或其他项目约定文件。

## 后端 Agent 与 DDD 流程

后端 Agent 开发新功能时，先判断是否涉及新的业务概念、用例、领域规则、聚合状态变化、领域事件、仓储语义、限界上下文关系或公开接口语义变化。只要涉及其中任一项，就使用插件内的 `ddd` 编排 skill，而不是直接从 Controller、数据库表或 CRUD 接口开始实现。

DDD 编排顺序如下：

1. 使用 `ddd` 作为入口，由它选择阶段和断点续跑方式。
2. 需要澄清业务边界、统一语言、子域、限界上下文或上下文映射时，进入 `ddd-strategic`。
3. 已有战略设计后，需要推导聚合、实体、值对象、领域服务、领域事件、仓储和不变量时，进入 `ddd-tactical`。
4. 已有战术设计后，进入 `ddd-implementation`，先识别目标项目约束、可执行规范、agent 指令、相似代码和测试门禁，再实现代码、接口适配、持久化和测试。

后端交接必须说明 DDD 产物和实现追溯关系：战略设计路径、战术设计路径、实现切片、接口文档路径、已运行测试和未使用 DDD 的理由。前端 Agent 只消费后端交付的接口契约，不直接参与后端领域建模。

## 默认并行模式

当用户点名本 skill、要求 agent 团队/并行工作，或任务本身需要前后端协同交付时，技术负责人必须先执行“强制调度门禁”，再让独立 Agent 或项目线程并行推进。若运行环境要求额外授权，先用一句话向用户请求授权。

- 主 Codex 只做最小事实盘点：项目地址、分支、命令、技术栈、接口契约来源和明显阻塞。
- 后端 Agent、前端 Agent 与测试 Agent 必须拥有不重叠的主要写入范围；任务包必须写明项目路径、职责边界、接口契约来源、验证命令、“先读取并遵循所属项目 `AGENTS.md`”和“不要 revert 他人修改”。后端任务包还必须说明：新业务功能先使用 `ddd` 编排，跳过 DDD 时必须写明理由。
- 主上下文只保留摘要，不粘贴大段源码、完整命令输出、全量文件清单或长接口文档；让责任 Agent 在各自上下文读取项目文件。
- 接口未知或需要变更时，后端 Agent 先判断是否属于新业务功能；属于新业务功能时先走 `ddd` 编排，再产出/更新接口契约。前端 Agent 可并行检查前端 API client 和页面结构，但不得猜测未文档化字段。
- 技术负责人只在需要集成决策时等待 Agent，收到结果后用交接摘要和定向 diff 做审查。

## 工作流程

1. 确认并向用户提供前端和后端项目地址。
2. 执行“强制调度门禁”，确认使用 sub-agent、后台线程、项目线程或任务包转交；没有可执行调度路径时停止在实现前。
3. 分别最小化检查两个项目的技术栈、包管理器、已有 API client、接口文档、测试和运行命令；这些检查优先由对应 Agent 或项目线程完成并回传摘要。
4. 阅读项目约定，例如 `AGENTS.md`、README、已有 OpenAPI 文件、路由定义、生成客户端和接口说明；实现侧项目约定由对应 Agent 读取并遵循。
5. 给出简短实施计划，包含后端任务、前端任务、接口契约工作、Agent/线程分工和验证命令，并记录 agent id、thread id 或任务包状态。
6. 并行启动或转交后端 Agent 与前端 Agent；如果接口形态未知，后端 Agent 先判断是否需要 DDD 编排并负责契约，前端 Agent 先做不依赖契约的本项目准备。
7. 前端开始真实接口对接前，必须确保存在可用接口文档。若没有现成接口文档，后端 Agent 必须先完成必要的 DDD 编排和后端实现设计，再根据后端路由、控制器、类型或 schema 生成接口文档，并保存到后端项目或清晰的共享文档位置。
8. 将接口文档路径交给前端 Agent，作为唯一接口契约来源。
9. 前端 Agent 根据接口文档实现页面和 API 调用。
10. 技术负责人审查两侧交接结果后，启动或复用独立测试 Agent 执行自动化验收：后端接口自动化、前端 service 映射测试、接口联调 smoke 脚本、浏览器 E2E。
11. 将失败项分派给对应角色修复，持续迭代，直到自动化验收通过或遇到缺凭据、服务不可用、测试数据不可写等真实阻塞。

## 接口文档规则

优先使用已有接口文档，例如 `openapi.yaml`、`openapi.yml` 或 `openapi.json`。如果后端项目已经有其他事实来源，例如类型化路由 schema、Swagger 装饰器、tRPC router、GraphQL schema 或生成客户端，则使用该来源，并导出或整理成前端可消费的接口文档。

如果没有可用接口文档，后端 Agent 必须在前端开发前创建文档。文档至少包含：

- 接口路径和 HTTP 方法。
- 路径参数、查询参数和请求体参数。
- 响应状态码和响应数据结构。
- 鉴权和权限要求。
- 错误响应结构。
- 新增或变更接口的请求示例和响应示例。

真实开发时，读取 `references/team-contract.md` 获取详细交接规则。

## 测试 Agent 规则

测试 Agent 默认在前后端实现完成后独立运行，避免主上下文承载长日志。测试流程顺序固定为：

1. 后端接口自动化：使用后端项目现有测试框架覆盖 Controller/API 路径、状态码、请求绑定、响应结构和关键异常。
2. 前端 service 映射测试：使用前端项目测试框架覆盖接口字段映射、枚举转换、payload 清洗、分页组树和命令端点封装。
3. 接口联调 smoke 脚本：对真实后端执行最小 CRUD + 清理闭环；缺少 token、服务或测试数据库时标记为阻塞，不得伪装通过。
4. 浏览器 E2E：使用项目内可重复执行的浏览器自动化脚本（如 Playwright/Cypress）驱动前端页面完成核心用户路径，验证鉴权、路由、页面状态、表单交互和真实 API 写入/清理闭环；缺少浏览器运行时、账号/验证码处理方案、前端服务、后端服务或测试数据库时标记为阻塞，不得伪装通过。

如果用户明确某个环境是稳定测试环境，例如 dev 环境，则最终通过必须基于该环境的标准启动、标准数据库、真实 smoke 和浏览器 E2E 结果。禁用迁移、临时 profile、手工 token 或本地替代环境只能作为定位手段，不能作为最终通过结论；共享数据库修复、Flyway `repair` 或 `out-of-order` 等写环境操作必须说明风险并获得确认。

Smoke 和 E2E 脚本默认应使用测试前缀数据，并在成功或失败后清理。测试通过后表里没有保留临时数据是预期；只有用户明确要求人工验收留痕时，才通过显式开关保留数据。

测试 Agent 交接只报告命令、通过/失败/阻塞状态、失败关键行和修改文件，不粘贴长日志。

### 浏览器 E2E 与可视化验收纪律

- 可视化浏览器操作和脚本化 E2E 必须分清：使用 in-app browser 或手动式自动操作可以作为可视化 smoke/定位手段，但不能替代项目内 Playwright/Cypress 等 E2E 脚本的通过结论。
- 用户要求“看到浏览器打开”时，优先运行 headed 模式的 E2E 脚本，让脚本打开真实浏览器并实际点击、输入、断言；若浏览器运行时缺失，优先配置使用本机已安装浏览器（如 Playwright `channel: 'chrome'`），需要下载安装时说明耗时和来源并按环境授权流程执行。
- 鉴权路径必须匹配真实用户路径：除非测试目标就是深链或 token 恢复，否则 E2E 不得通过注入 localStorage/token、伪造请求或直接访问受保护页面来宣称通过；应从登录页开始，通过导航菜单或页面入口进入目标功能。
- 遇到验证码、二次确认等真实交互时，遵循浏览器安全规则先获得用户授权。获得授权后，E2E 脚本应实现可重复的处理方式（例如识别简单验证码、使用测试环境公开的测试开关，或明确标记为阻塞）；不要把一次人工填写当作自动化通过。
- 脚本断言要以稳定信号为准：URL、可访问名称、`data-testid`、行属性、接口返回数据、数据库/API 可见结果。Toast 可作为辅助观察，不宜作为唯一断言，尤其要避免重复 toast 导致 strict mode 偶发失败。
- E2E 失败时不要降低验证标准。保留失败关键行、trace/screenshot 路径或最后一次识别/输入记录，修复脚本或实现后重跑；只有脚本命令返回通过，才能在最终结果中写“浏览器 E2E 通过”。

## 开发纪律

遵循项目已有约定和工具。前端 Agent 和后端 Agent 写代码前必须读取所属项目的 `AGENTS.md`，并按其中规范实现、测试和交接；不要随意引入新的 API 风格、客户端生成器、状态库或测试框架，除非项目已有明显方向或用户明确要求。

采用契约优先的对接方式：

- 后端变更必须同步更新接口文档。
- 前端必须消费文档中的接口契约，不要猜字段、路径或错误结构。
- 发现契约不一致时，记录问题、分派责任角色，并从源头修正。
- Agent 交接只报告改动文件、关键行为、验证结果和阻塞；长日志、完整源码和大段接口内容留在文件或工具输出中。

## 最终回复

总结时必须包含：

- 使用的前端项目地址和后端项目地址。
- 调度方式、agent id/thread id，或无法调度时输出的任务包与阻塞原因。
- 生成或更新的接口文档位置。
- 后端 DDD 产物位置，或未使用 DDD 的理由。
- 后端完成内容和前端完成内容。
- 已运行的测试、自动化 smoke、浏览器 E2E 结果和集成检查。
- 剩余阻塞或需要用户处理的事项。
