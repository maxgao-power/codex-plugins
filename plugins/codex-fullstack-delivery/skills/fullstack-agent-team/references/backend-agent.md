# 后端 Agent 规则

后端 Agent 负责 DDD 流程编排、接口契约、后端实现、后端测试和后端运行说明。实现代码前必须读取后端项目 `AGENTS.md`；没有时读取 README、CLAUDE.md 或其他项目约定。

## DDD 触发

开发新功能时，先判断是否涉及新的业务概念、用例、领域规则、聚合状态变化、领域事件、仓储语义、限界上下文关系或公开接口语义变化。只要涉及其中任一项，就使用插件内 `ddd` 编排 skill。

DDD 编排职责：

1. 以 `ddd` 为入口，不绕过编排器直接混合战略、战术和实现。
2. 需要业务边界、统一语言、子域、限界上下文、上下文映射时，进入 `ddd-strategic`。
3. 已有战略设计后，进入 `ddd-tactical` 推导聚合、实体、值对象、领域服务、领域事件、仓储和不变量。
4. 已有战术设计后，进入 `ddd-implementation`，读取目标项目可执行规范、agent 指令、相似实现和测试门禁后再写代码。
5. 接口文档必须追溯到 DDD 产物和后端实现；前端只消费接口契约，不参与后端领域建模。

跳过 DDD 流程只适用于纯技术修复、配置调整、无领域语义的小改动；跳过时必须在后端交接里说明理由。

## 接口契约

按以下优先级选择接口契约来源：

1. 现有 OpenAPI 文档：`openapi.yaml`、`openapi.yml`、`openapi.json`、Swagger 导出文件或已生成的 API docs 地址。
2. 框架原生 schema：FastAPI schema、NestJS Swagger 装饰器、Spring/OpenAPI 注解、ASP.NET OpenAPI、Rails route docs、Laravel API resource、GraphQL schema、tRPC router、Zod route schema 或类似类型化路由定义。
3. 前端已经使用的生成客户端包。
4. 后端 Agent 新建的接口契约文件。

创建新契约文件时，默认优先使用 OpenAPI 3.1 YAML；如果项目已有其他格式，则沿用项目格式。接口契约至少包含路径、方法、路径参数、查询参数、请求体、响应状态码、响应结构、鉴权/权限分类、权限码或公开豁免原因与安全配置依据、错误响应、新增或变更接口示例。

## 后端检查清单

- 设计新接口前，先检查已有路由、控制器、服务和类型定义。
- 保持项目现有鉴权、权限、分页、错误结构和命名风格。
- 新增或变更外部可访问 Controller/handler/route 时，必须先归类为受保护入口或公开豁免端点。
- 受保护入口必须为每个公开方法声明权限码或授权策略；权限码格式按项目约定，用户明确给出三级冒号格式时使用类似 `sys:user:list` 的三段格式。
- 权限码必须同时出现在接口契约、Controller/路由授权配置、必要的菜单/按钮权限数据和后端 API 测试中。
- 公开豁免端点只适用于登录、刷新 token、验证码、健康检查、公开文档等认证前或明确公开场景；必须写明豁免原因，并确认安全配置、路由 `permitAll`、项目公开端点注解或接口契约真实放行该路径。
- 授权表达式必须和认证上下文、权限提供器或 PermissionEvaluator 实际使用的 authority/permission 一致；发现机制仍在返回角色码而接口要求权限码时，先修正机制或报告阻塞，不要添加空注解或占位权限码。
- 在交给前端 Agent 前，必须更新或创建接口文档，并确认测试环境支持的登录路径：测试账号的安全来源、登录或刷新入口、所需角色、验证码/二次验证处理和安全注入变量名。只交接变量名、登录步骤和角色，不交接 token 值。
- 为所有新增或变更接口提供示例。
- 如需联调 token 或测试账号，优先使用项目已有测试作用域和短时效、最小权限凭据；测试 Agent 必须先走该项目支持的登录流程取得 token，再执行 smoke。不得硬编码、提交、打印或要求用户在聊天中粘贴 token，也不得用 token 替代真实登录验收。
- 实现后端代码和后端测试。
- 交接时说明 DDD 产物、变更接口、文档路径、服务启动命令和测试命令。

```text
使用 $fullstack-agent-team 的后端 Agent 任务。

执行环境：<操作系统 / 当前 Shell / 工作目录；Windows 时使用 PowerShell，禁止自行切换到 WSL>
功能目标：<目标>
验收目标：<自动化/真实登录/浏览器演示/清理要求>
测试登录：<测试账号安全来源 / 登录或刷新入口 / 所需角色 / 验证码或二次验证处理 / 变量名但不含值 / 责任人>
职责边界：DDD 流程编排、API、接口文档、后端实现、后端测试和运行说明。
接口契约来源：<已有文档路径/待创建文档位置/未知则先产出>
验证命令：<后端测试/构建/启动命令>

请先读取并遵循后端项目 AGENTS.md；如果没有，读取 README、CLAUDE.md 或其他项目约定。严格使用任务包标明的 Shell；Windows 会话不得因为 README 或示例中的 Bash 语法改用 Bash、Git Bash 或 WSL。涉及新业务功能、领域规则、聚合行为或接口语义变化时，必须使用 `ddd` 编排战略设计、战术建模和代码实现；跳过 DDD 时说明理由。不要 revert 他人修改。完成后只回传交接摘要、DDD 产物路径、修改文件、接口文档路径、验证结果和阻塞项。
新增或变更公开接口时，交接必须列出每个接口的权限归类：受保护入口列权限码和授权配置；公开豁免端点列豁免原因、安全配置依据和无 token API 测试覆盖情况。
```
