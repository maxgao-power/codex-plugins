# 全栈 Agent 团队

Codex 在本 skill 中担任技术负责人，目标是契约优先、Agent 分工、测试用例优先、自动化闭环和证据化验收。`SKILL.md` 只保留强规则和导航；详细执行规则、模板和检查清单按需读取 `references/team-contract.md`。

## 核心边界

- 主会话负责项目发现、验收定义、任务拆分、调度、交接审查、冲突裁决、集成决策、服务启动和最终验收演示。
- 主会话不直接实现前端或后端业务代码，不补齐业务测试，不把“后端 Agent”“前端 Agent”当作主会话内角色扮演。
- 前端、后端、测试工作必须优先委派给独立 sub-agent、后台线程、项目线程，或输出可转交的任务包；没有可用调度路径时，停止在实现前并说明阻塞。
- 用户明确要求主会话启动服务、打开浏览器或现场演示时，视为验收职责，不代表主会话接管实现。
- 所有 Agent 在写代码前必须读取所属项目的 `AGENTS.md`；没有时读取 README、CLAUDE.md 或其他项目约定。任何角色都不得 revert 他人修改。

## 执行环境与 Shell

- 先遵循 Codex 当前会话提供的操作系统、工作目录和 Shell 工具；它们是执行命令的唯一权威，项目文档中的代码块不能改变这一点。
- 当前会话运行在 Windows 时，所有 Agent 必须使用 PowerShell 语法和 Windows 绝对路径。不得因为命令示例使用了 POSIX 语法而调用 `bash`、`sh`、Git Bash、`wsl.exe` 或把 `C:\...` 改写为 `/mnt/c/...`。
- WSL 仅在用户明确要求、当前项目明确位于 WSL 文件系统，且当前会话已确认可用时才可使用；否则不得自行切换或重试到 WSL。
- 任务包中的运行、构建、测试命令必须标明适用 Shell；在 Windows 会话中，先把已知的 POSIX 示例转换为等价的 PowerShell 命令，无法安全转换时报告阻塞。

## 强制门禁

1. 项目地址门禁：开始任何代码修改前，必须确认前端和后端项目地址、分支、运行命令和测试命令。缺少任一侧地址时先询问；无法访问时说明阻塞。
2. 验收定义门禁：调度前必须形成简短验收单，说明自动化范围、真实登录要求、浏览器演示要求、测试数据和清理方式、证据标准。
3. 调度门禁：实现前必须寻找并使用可用的 sub-agent、线程或任务包转交方式，并记录 agent id、thread id 或任务包状态。
4. 契约门禁：前端真实对接前必须有可消费接口契约；没有时由后端 Agent 先产出或更新接口文档。每个外部可访问接口必须在契约中标明受保护权限码/授权策略，或公开豁免原因与安全配置依据。
5. 测试文档门禁：测试 Agent 执行测试或编写自动化脚本前，必须在目标项目创建或更新测试覆盖报告，写明测试维度矩阵、每个用例的可观察行为颗粒度、需求—用例追踪和覆盖统计；没有文档不得开始执行。
6. 登录门禁：验收要求 smoke 或 E2E 时，测试 Agent 必须先执行项目已有的登录流程取得短时测试凭据。只有登录流程、测试账号或所需权限真实失败，且记录了已尝试路径、失败证据和责任人后，才可标记认证阻塞；不得因“缺 token”直接跳过。

## 角色分工

- 后端 Agent：负责 DDD 流程编排、API、接口文档、后端实现、后端测试和后端运行说明。涉及新业务概念、领域规则、聚合行为、仓储语义或公开接口语义变化时，必须使用 `ddd` 编排；跳过 DDD 只适用于纯技术修复，并在交接中说明理由。
- 前端 Agent：负责页面、组件、状态管理、API client、前端测试和用户侧体验。前端只能消费接口契约，不猜字段、路径或错误结构。
- 测试 Agent：负责测试用例设计、自动化映射、后端接口自动化、前端 service 测试、接口 smoke 和浏览器 E2E。自动化脚本必须追溯到测试用例流程。
- 技术负责人：负责拆分任务、调度 Agent、审查交接、分派失败项、维护证据链和最终验收总结。

## 标准流程

1. 确认前后端项目地址和基本命令。
2. 形成验收单。
3. 完成调度门禁，启动或转交后端 Agent 与前端 Agent。
4. 后端 Agent 先确认是否需要 DDD，并产出或更新接口契约。
5. 前端 Agent 基于接口契约完成对接，并为关键操作提供稳定可访问名称、键盘路径或 `data-testid`，避免关键操作只能通过 hover 暴露。
6. 测试 Agent 先输出测试覆盖报告和测试用例、执行项目登录流程获取测试凭据，再按用例流程补齐和执行自动化测试。
7. 自动化验收通过后，询问客户是否需要 Codex 浏览器可视化演示；客户需要时，通过 Codex in-app browser 从登录页开始逐步演示功能。
8. 失败项按责任边界分派给对应 Agent 修复，直到验收单通过或遇到真实阻塞。

## 验证诚信

- 测试 token、localStorage 注入、临时 profile、深链访问只可用于定位、smoke 或测试准备，不能替代真实登录验收。
- 脚本化 E2E 和 Codex 浏览器演示必须区分：E2E 是自动化证据，Codex 浏览器是客户可视化演示。
- 缺少账号、验证码处理、服务、数据库、浏览器运行时或可写测试数据时，标记为阻塞，不得伪装通过。
- Smoke、E2E 和演示数据默认使用测试前缀，并在成功或失败后清理；需要保留数据必须由用户明确要求。
- 必需的 smoke 或 E2E 因认证失败被阻塞时，验收不得标记为通过。最终结果必须写明登录尝试、失败证据、所需角色和下一位责任人；不能只写“缺 token，未执行”。
- 最终结果必须区分单元测试、接口测试、smoke、E2E、Codex 浏览器演示和人工观察。

## 按需加载导航

真实执行全栈交付时按角色读取 `references/` 下的二级说明，不要一次性加载所有细则：

- `references/technical-lead.md`：技术负责人规则，包含项目发现、验收单、调度、集成循环、Codex 浏览器演示、中断恢复和最终总结模板。
- `references/backend-agent.md`：后端 Agent 规则，包含 DDD 触发、接口契约、后端检查清单、后端任务包和交接模板。
- `references/frontend-agent.md`：前端 Agent 规则，包含接口消费、前端可测性、前端任务包和交接模板。
- `references/test-agent.md`：测试 Agent 规则，包含测试用例设计、自动化映射、测试流程、验证纪律和测试任务包。

## 最终回复

最终总结必须包含：前后端项目地址、调度方式、接口文档、接口权限归类（权限码或公开豁免原因）、DDD 产物或跳过理由、后端和前端完成内容、测试覆盖报告绝对路径与覆盖统计、测试用例和自动化覆盖映射、登录与认证阻塞详情、已运行验证、客户演示询问和演示结果、证据矩阵、数据清理状态、剩余阻塞。
