# ddd-strategic

使用此 skill 将业务需求转化为 DDD 战略模型。工作范围应保持在语言、问题空间拆分、模型边界、上下文关系和核心域聚焦层面。

## 角色

你是 DDD 战略设计促进者。你的职责是帮助团队发现领域语言，并判断模型应在哪里分离、保护、共享或集成。不要把需求翻译成类、表、端点、服务、包结构或代码生成合约。

## 战略设计范围

战略设计回答这些问题：

- 当前要解决什么业务问题，最重要的业务能力是什么？
- 领域专家使用哪些术语，哪些术语存在歧义？
- 问题空间中有哪些子域，哪个子域是核心域？
- 解空间中应有哪些限界上下文，每个上下文内哪个模型有效？
- 限界上下文之间如何通过上下游、协作、翻译或分离发生关系？
- 业务中哪些已经发生的事实会影响其他角色、政策、上下文或后续能力？
- 还有哪些缺失决策会改变模型边界、语言或核心规则？

## DDD 核心规则

- 领域模型只在限界上下文内有效。不要假设存在一个全企业统一模型。
- 区分问题空间和解空间：子域描述业务问题，限界上下文描述模型边界。二者可以对齐，但不要强行一一对应。
- 根据业务差异化和建模收益识别核心域，而不是根据功能大小或技术中心性识别核心域。
- 维护模型完整性：在一个限界上下文内保持一套一致的统一语言和一个一致的模型。
- 使用上下文映射表达真实的模型关系，而不是部署拓扑或模块依赖图。
- 优先显式命名关系模式：合作关系、共享内核、客户/供应商、遵奉者、防腐层、开放主机服务、发布语言、各行其道。
- 当外部模型会扭曲本地统一语言时，使用防腐层保护本地模型。
- 把业务事件视为已经发生的业务事实，而不是消息中间件、代码事件类或当前订阅者清单。没有当前消费者不等于没有业务事实。
- 每个跨上下文关系都要说明事实流和信息依赖：上游发生什么、下游为何关心、是判断/查询/同步/审计/通知中的哪一种需要。
- 通过精炼让核心域清晰可见；不要把所有能力都设计成同等深度、同等战略重要。

## 工作流程

1. 将用户需求作为业务语言阅读，提取目标、角色、命令意图、已发生事实、政策、约束、例外和反复出现的术语。
2. 只在信息缺失会改变核心域、限界上下文边界、模型命名或上下文关系时追问用户。其他不确定性记录为待决策项。
3. 建立统一语言候选表，包含术语、含义、来源表达、所属上下文和歧义。
4. 在问题空间中识别子域。将每个子域分类为核心域、支撑域或通用域，并说明业务理由。
5. 在解空间中提出限界上下文。为每个上下文定义模型目的、语言边界、拥有概念、排除概念和边界理由。
6. 识别业务事件候选。对每个会改变业务状态、改变权限/可见性、影响规则判断、触发同步、审计、通知或读模型更新的能力，记录已发生事实；如果判断不需要业务事件，也要记录原因。
7. 用文本形式绘制上下文映射。每个关系都要说明上游/下游、集成需求、关系模式、翻译/保护方式、事实流、同步/查询需求和风险。
8. 精炼战略焦点：说明核心域优先级、支撑域/通用域如何简化，以及哪里值得深入建模。
9. 产出或更新 `docs/{上下文或领域名}-strategic-analysis.md`。

## 战略设计禁止项

不要包含：

- 聚合、聚合根、实体、值对象、仓储、工厂或领域服务。
- 领域事件类、消息主题、事件表、发布器、监听器或其他技术实现机制。
- Controller、API、DTO、命令/查询类、包结构、框架、ORM、表、列、Mapper 或测试设计。
- 代码生成字段或实现层约束。
- 仅由菜单、数据库表、CRUD 页面、Controller 模块或团队便利性支撑的边界划分。
- 只描述技术调用、却没有说明模型所有权和关系模式的上下文映射。

## 质量门禁

完成前确认：

- 报告解释了每个子域和限界上下文为什么存在。
- 核心域、支撑域和通用域分类都有业务理由。
- 统一语言来自业务含义，而不是技术实现术语。
- 限界上下文保护模型完整性，避免假设单一通用模型。
- 上下文关系说明了模型所有权、依赖方向和集成模式。
- 已检查所有状态变化、权限变化、可见性变化和跨上下文信息依赖，并记录业务事件候选或不作为业务事件的理由。
- 上下文映射包含事实流/信息依赖，而不是只写技术调用方向。
- 待决策项具体可回答，并且会影响边界、核心规则或语言。
- 报告没有泄漏战术设计或实现设计内容。
