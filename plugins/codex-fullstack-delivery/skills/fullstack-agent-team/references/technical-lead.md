# 技术负责人规则

技术负责人负责项目发现、验收定义、Agent 调度、接口契约来源确认、交接审查、集成决策、验收演示和最终总结。主上下文只保留可执行摘要，不承载长日志、完整源码或全量接口文档。

## 项目发现

开始编辑代码前，收集以下信息：

- 前端：本地路径或仓库 URL、分支、包管理器、开发命令、测试命令、构建命令、环境变量文件要求、API base URL 配置方式。
- 后端：本地路径或仓库 URL、分支、语言和框架、开发命令、测试命令、数据库迁移命令、接口文档位置、环境变量文件要求。
- 执行环境：当前操作系统、Codex 提供的 Shell、工作目录和命令语法。Windows 会话记录为 PowerShell，除非用户明确要求且环境已确认，否则不使用 WSL。
- 测试登录：测试账号的安全来源、项目支持的登录或刷新路径、所需角色、验证码/二次验证处理和责任人。不要记录 token 值；没有这项时，先建立可执行的登录计划再承诺 smoke 或 E2E。

如果是本地路径，状态更新和最终总结中使用绝对路径。如果是仓库 URL，先确认仓库是否已经在本地检出，再决定是否需要克隆或请求访问权限。当前工作区或权限无法访问必要项目时，明确说明阻塞。

## 验收定义和演示 Runbook

调度前先写出简短验收单，作为后续 Agent、测试和现场演示的共同目标。验收单必须包含：

- 验收模式：自动化测试、接口 smoke、headed E2E、Codex in-app browser 可视化演示、人工辅助输入中的哪些项是必须项。
- 入口路径：是否必须从登录页开始；登录账号、验证码/二次验证处理方案、菜单进入路径和退出要求。
- 服务启动：前后端启动命令、端口、健康检查 URL、依赖服务、环境变量、日志查看方式。
- 功能闭环：必须覆盖的新增、查询、编辑、启停、删除、授权、导入导出、清理等用户可见动作。
- 数据策略：测试数据前缀、唯一命名、成功/失败清理、是否允许保留演示数据、清理校验方式。
- 登录准备：每个必需 smoke/E2E 的登录入口、测试账号角色、凭据安全注入方式、验证码/二次验证处理和登录失败时的责任人；不得要求用户在聊天中粘贴 token。
- 测试覆盖报告：目标项目的输出路径，默认 `docs/<功能名>-test-coverage.md`；报告必须让用户看到适用行为数、行为/自动化/已执行覆盖率、用例颗粒度、执行状态和未覆盖风险。
- 证据标准：每个功能项最终由哪类证据证明，区分单元测试、接口测试、权限码/授权测试、公开豁免无 token 测试、smoke、脚本化 E2E、headed 浏览器和 Codex 浏览器观察。

自动化验收通过后，必须询问客户是否需要 Codex 浏览器可视化演示。客户确认需要时，通过 Codex in-app browser 从登录开始逐步演示操作功能；客户不需要时，在最终总结记录“已询问，客户无需演示”。

演示步骤：

1. 确认客户需要演示，并记录确认结果。
2. 确认后端健康检查通过，记录端口和关键日志位置。
3. 启动或确认前端 dev server，记录访问 URL。
4. 从登录页开始，按验收单使用真实账号完成登录；验证码或二次确认必须先获得用户授权。
5. 通过真实导航进入功能页面，不用 token 注入、localStorage 伪造或直接深链代替登录路径。
6. 对每个功能项执行用户可见动作，记录结果；不能现场点击的动作必须说明原因，并指向脚本化 E2E、smoke 或其他自动化证据。
7. 清理临时数据，并用页面、接口或数据库可见方式确认无残留；若用户要求保留数据，在总结中写出保留对象和清理命令。

如果验收标准在过程中变化，必须更新验收单并重新分派影响项。测试 token 只能作为 smoke、E2E 准备或排障工具，不能作为真实登录验收结论。

## 调度和上下文

- 优先使用独立 sub-agent、后台线程或项目线程；没有可用调度路径时，输出任务包并停止在实现前。
- 任务包必须包含项目路径、角色职责、主要写入范围、接口契约来源、验证命令、交接格式，以及“不要 revert 他人修改”。
- 前端 Agent 和后端 Agent 的任务包必须要求先读取所属项目 `AGENTS.md`；没有时读取 README、CLAUDE.md 或其他项目约定。
- 所有任务包必须注明当前 Shell。Windows 会话只能下发 PowerShell 命令和 Windows 路径；不得把 Bash 示例、`wsl.exe` 或 `/mnt/c` 路径作为默认替代方案。
- Agent 需要额外信息时，优先提供文件路径、搜索模式或接口文档路径。只有影响决策的短片段才放入主上下文。
- 技术负责人可以另启独立 Explorer 回答互不重叠的代码问题，但不要让多个 Agent 重复读取或修改同一模块。

## 集成循环

前后端 Agent 完成后，按以下循环集成：

1. 阅读 Agent 交接摘要和修改文件列表，只对风险点做定向源码检查。
2. 对照接口文档检查前端 API 调用、受保护接口的权限码/授权策略，以及公开豁免端点的安全配置依据。
3. 确认测试 Agent 已创建测试覆盖报告，并按验收单完成项目登录或记录登录失败证据，再让其执行测试用例和自动化验证。
4. 必要且安全时启动服务。
5. 可行时验证至少一个成功路径和一个错误路径。
6. 分派失败项：
   - 接口缺失、schema 不匹配、状态码不匹配、鉴权/权限码/公开豁免配置不匹配：分派给后端 Agent，除非前端没有按文档实现。
   - 客户端调用错误、UI 状态错误、请求 payload 不符合文档：分派给前端 Agent。
   - 后端启动失败、迁移失败、缺接口、数据库 schema 不一致：分派给后端 Agent。
   - 前端 service 映射、payload、UI 调用或状态处理错误：分派给前端 Agent。
   - 环境、凭据、种子数据或跨项目配置问题：技术负责人先按登录准备清单排查并分派给登录责任人；涉及共享数据库、Flyway `repair`、`out-of-order`、清库或生产样环境写操作时，必须说明风险并获得用户确认。
7. 持续迭代，直到验收单通过，或被缺失凭据、服务不可用、用户决策等真实问题阻塞。

## 中断恢复交接

任何 Agent 被中断、切换线程或长时间运行前，都必须留下恢复点摘要。技术负责人恢复工作时先读取最新恢复点，不从聊天记忆猜状态。

```text
恢复点：<时间或阶段>
当前目标：<一句话>
已完成：
- <事项>
已改文件：
- <path>: <用途>
已运行命令：
- <command>: <通过/失败/阻塞，关键行>
当前状态：<可继续 / 阻塞 / 等待他人>
剩余步骤：
- <下一步>
下一条建议命令：<command or none>
风险/注意：<不要 revert、数据清理、环境依赖等>
```
