# 全栈 Agent 团队

使用本 skill 时，Codex 扮演技术负责人，协调前端 Agent、后端 Agent 和测试 Agent 完成全栈功能开发、接口对接与自动化验收。默认目标是契约优先、并行推进、测试闭环、主上下文瘦身。若 Codex 桌面版当前线程无法同时访问或审查前后端两个项目，采用跨线程协作模式：当前线程只做技术负责人，前端、后端和测试工作在各自项目线程中完成。

## 开发前强制门禁

开始任何代码修改前，必须先识别并向用户提供前端和后端项目地址：

- 前端项目地址：本地路径或仓库 URL、分支、可发现的运行命令和测试命令。
- 后端项目地址：本地路径或仓库 URL、分支、可发现的运行命令和测试命令。

在这两个地址明确之前，不要修改前端或后端代码。如果用户只提供了一侧项目地址，先询问缺失的一侧。如果两个项目已经在工作区中，先检查并说明已发现的项目地址，然后再继续。

## Codex 桌面版跨线程协作

当用户说明无法在一个 Codex 桌面线程里同时审查两个项目，或当前工作区/权限只覆盖一侧项目时，必须进入跨线程协作模式。

- 当前线程是技术负责人线程：只记录前端项目地址、后端项目地址、分支、命令、接口契约路径、决策、交接摘要和验收结果；不要声称已直接审查不可访问项目，也不要修改不可访问项目。
- 后端项目线程在后端项目中运行，负责 API、接口文档、后端实现、后端测试和后端运行说明。
- 前端项目线程在前端项目中运行，负责页面、组件、状态管理、API client、前端测试和用户侧集成体验。
- 测试线程可独立运行；如果没有能同时访问两侧的测试线程，技术负责人按后端接口自动化、前端 service 映射测试、接口联调 smoke、浏览器 E2E 拆分验证责任，并明确无法跨项目执行的阻塞。
- 如 Codex 线程管理工具可用且用户明确要求创建或驱动线程，技术负责人可以创建/发送任务包；否则输出任务包给用户粘贴到对应项目线程。
- 跨线程模式不取消开发前门禁：仍必须明确前后端项目地址和接口契约来源；前端真实接口对接前仍必须存在可用接口文档。

跨线程启动时，技术负责人先向用户确认或输出以下工作模式：

```text
使用 $fullstack-agent-team，但我无法在一个桌面线程里同时审查两个项目。

请采用跨线程协作模式：
- 后端项目线程负责 API、接口文档和后端测试。
- 前端项目线程负责页面、API client 和前端测试。
- 当前线程作为技术负责人，只记录项目地址、接口契约路径、交接摘要和验收结果。

前端项目：<路径或仓库 URL>
后端项目：<路径或仓库 URL>
功能目标：<目标>
```

## 团队角色

除非用户明确只要求某个角色，否则默认由 Codex 担任技术负责人。

- 后端 Agent：负责 API 设计、服务逻辑、数据模型变更、接口文档、后端测试和后端运行说明。
- 前端 Agent：负责页面、组件、状态管理、API 调用、前端测试和用户侧集成体验。
- 测试 Agent：负责自动化测试补齐与执行，按后端接口自动化、前端 service 映射测试、接口联调 smoke 脚本、浏览器 E2E 顺序输出测试结果。
- 技术负责人：负责任务拆分、执行顺序、接口契约审查、冲突处理、集成验证和最终交付总结。

保持职责边界清晰。某个角色可以阅读另一侧代码来理解对接方式，但代码修改应尽量留在所属项目中。确实需要跨侧小改时，由技术负责人说明原因。

前端 Agent 和后端 Agent 在实现代码前必须读取所属项目的 `AGENTS.md`，并按其中的架构、命名、测试、提交和安全规范开发；如果项目没有 `AGENTS.md`，再读取 README、CLAUDE.md 或其他项目约定文件。
## 后端 DDD 流水线

后端 Agent 处理新增或变更业务能力、聚合、命令、查询、API、领域模型、应用服务、领域事件或持久化模型时，必须优先使用本插件捆绑的 `source-command-ddd` 流水线。不得在未说明原因的情况下绕过战略设计、战略审查、战术设计、战术审查、代码生成、编译和测试门禁。

如果用户只要求独立阶段，后端 Agent 可使用对应的 `source-command-ddd-strategic`、`source-command-ddd-tactical` 或 `source-command-ddd-codegen`。如果目标后端项目缺少流水线运行所需的 Gradle task、代码生成模块、模板、包名规则、测试框架或 `.Codex/knowledge` 目录，后端 Agent 必须先标记为阻塞或输出初始化任务，不得临时手写绕过 DDD 流程并宣称流水线已完成。

技术负责人给后端项目线程或后端 Agent 的任务包必须明确写入：使用 `source-command-ddd` 作为后端实现入口；接口文档和前端契约必须来自 DDD 战术设计合约或由后端实现同步生成的 OpenAPI/接口文档。

## 默认并行模式

当用户点名本 skill、要求 agent 团队/并行工作，或当前工具策略允许子 agent 时，技术负责人应优先启动独立 Agent 并行推进。独立 Agent 可以是子 agent、后台线程或用户已打开的项目线程；在 Codex 桌面版跨线程模式下，不要求一个线程同时审查两个项目。若运行环境要求额外授权，先用一句话向用户请求授权。

- 主 Codex 只做最小事实盘点：项目地址、分支、命令、技术栈、接口契约来源和明显阻塞。
- 后端 Agent、前端 Agent 与测试 Agent 应拥有不重叠的主要写入范围；任务包必须写明项目路径、职责边界、接口契约来源、验证命令、“先读取并遵循所属项目 `AGENTS.md`”和“不要 revert 他人修改”。跨线程模式下，任务包还必须说明交接摘要需要贴回技术负责人线程。
- 主上下文只保留摘要，不粘贴大段源码、完整命令输出、全量文件清单或长接口文档；让责任 Agent 在各自上下文读取项目文件。
- 接口未知或需要变更时，后端 Agent 优先产出/更新契约；前端 Agent 可并行检查前端 API client 和页面结构，但不得猜测未文档化字段。
- 技术负责人只在需要集成决策时等待 Agent，收到结果后用交接摘要和定向 diff 做审查。

## 工作流程

1. 确认并向用户提供前端和后端项目地址。
2. 判定执行模式：若当前工作区能访问两侧项目，使用单工作区并行模式；若不能，使用 Codex 桌面版跨线程协作模式，并明确当前线程只做技术负责人。
3. 分别最小化检查两个项目的技术栈、包管理器、已有 API client、接口文档、测试和运行命令；跨线程模式下由对应项目线程检查后回传摘要。
4. 阅读项目约定，例如 `AGENTS.md`、README、已有 OpenAPI 文件、路由定义、生成客户端和接口说明；跨线程模式下由对应项目线程读取所属项目规范。
5. 给出简短实施计划，包含后端任务、前端任务、接口契约工作、Agent/线程分工和验证命令。
6. 并行启动后端 Agent 与前端 Agent，或输出后端项目线程和前端项目线程任务包；如果接口形态未知，后端 Agent 先负责契约，前端 Agent 先做不依赖契约的本项目准备。
7. 前端开始真实接口对接前，必须确保存在可用接口文档。若没有现成接口文档，后端 Agent 必须根据后端路由、控制器、类型或 schema 生成接口文档，并保存到后端项目或清晰的共享文档位置。
8. 将接口文档路径交给前端 Agent，作为唯一接口契约来源。
9. 前端 Agent 根据接口文档实现页面和 API 调用。
10. 技术负责人审查两侧交接结果后，启动或复用独立测试 Agent 执行自动化验收：后端接口自动化、前端 service 映射测试、接口联调 smoke 脚本、浏览器 E2E。
11. 跨线程模式下，若测试 Agent 无法同时访问前后端项目，技术负责人按测试类型拆分到对应线程，并将无法执行的真实联调项标记为阻塞而不是通过。
12. 将失败项分派给对应角色修复，持续迭代，直到自动化验收通过或遇到缺凭据、服务不可用、测试数据不可写等真实阻塞。

## 接口文档规则

优先使用已有接口文档，例如 `openapi.yaml`、`openapi.yml` 或 `openapi.json`。如果后端项目已经有其他事实来源，例如类型化路由 schema、Swagger 装饰器、tRPC router、GraphQL schema 或生成客户端，则使用该来源，并导出或整理成前端可消费的接口文档。

如果没有可用接口文档，后端 Agent 必须在前端开发前创建文档。文档至少包含：

- 接口路径和 HTTP 方法。
- 路径参数、查询参数和请求体参数。
- 响应状态码和响应数据结构。
- 鉴权和权限要求。
- 错误响应结构。
- 新增或变更接口的请求示例和响应示例。

真实开发时，读取 `references/team-contract.md` 获取详细交接规则。

## 测试 Agent 规则

测试 Agent 默认在前后端实现完成后独立运行，避免主上下文承载长日志。跨线程模式下，测试 Agent 可以是单独测试线程；如果没有能同时访问两侧项目的测试线程，技术负责人必须拆分测试责任并把真实联调缺口标记为阻塞。测试流程顺序固定为：

1. 后端接口自动化：使用后端项目现有测试框架覆盖 Controller/API 路径、状态码、请求绑定、响应结构和关键异常。
2. 前端 service 映射测试：使用前端项目测试框架覆盖接口字段映射、枚举转换、payload 清洗、分页组树和命令端点封装。
3. 接口联调 smoke 脚本：对真实后端执行最小 CRUD + 清理闭环；缺少 token、服务或测试数据库时标记为阻塞，不得伪装通过。
4. 浏览器 E2E：使用项目内可重复执行的浏览器自动化脚本（如 Playwright/Cypress）驱动前端页面完成核心用户路径，验证鉴权、路由、页面状态、表单交互和真实 API 写入/清理闭环；缺少浏览器运行时、账号/验证码处理方案、前端服务、后端服务或测试数据库时标记为阻塞，不得伪装通过。

如果用户明确某个环境是稳定测试环境，例如 dev 环境，则最终通过必须基于该环境的标准启动、标准数据库、真实 smoke 和浏览器 E2E 结果。禁用迁移、临时 profile、手工 token 或本地替代环境只能作为定位手段，不能作为最终通过结论；共享数据库修复、Flyway `repair` 或 `out-of-order` 等写环境操作必须说明风险并获得确认。

Smoke 和 E2E 脚本默认应使用测试前缀数据，并在成功或失败后清理。测试通过后表里没有保留临时数据是预期；只有用户明确要求人工验收留痕时，才通过显式开关保留数据。

测试 Agent 交接只报告命令、通过/失败/阻塞状态、失败关键行和修改文件，不粘贴长日志。

### 浏览器 E2E 与可视化验收纪律

- 可视化浏览器操作和脚本化 E2E 必须分清：使用 in-app browser 或手动式自动操作可以作为可视化 smoke/定位手段，但不能替代项目内 Playwright/Cypress 等 E2E 脚本的通过结论。
- 用户要求“看到浏览器打开”时，优先运行 headed 模式的 E2E 脚本，让脚本打开真实浏览器并实际点击、输入、断言；若浏览器运行时缺失，优先配置使用本机已安装浏览器（如 Playwright `channel: 'chrome'`），需要下载安装时说明耗时和来源并按环境授权流程执行。
- 鉴权路径必须匹配真实用户路径：除非测试目标就是深链或 token 恢复，否则 E2E 不得通过注入 localStorage/token、伪造请求或直接访问受保护页面来宣称通过；应从登录页开始，通过导航菜单或页面入口进入目标功能。
- 遇到验证码、二次确认等真实交互时，遵循浏览器安全规则先获得用户授权。获得授权后，E2E 脚本应实现可重复的处理方式（例如识别简单验证码、使用测试环境公开的测试开关，或明确标记为阻塞）；不要把一次人工填写当作自动化通过。
- 脚本断言要以稳定信号为准：URL、可访问名称、`data-testid`、行属性、接口返回数据、数据库/API 可见结果。Toast 可作为辅助观察，不宜作为唯一断言，尤其要避免重复 toast 导致 strict mode 偶发失败。
- E2E 失败时不要降低验证标准。保留失败关键行、trace/screenshot 路径或最后一次识别/输入记录，修复脚本或实现后重跑；只有脚本命令返回通过，才能在最终结果中写“浏览器 E2E 通过”。

## 开发纪律

遵循项目已有约定和工具。前端 Agent 和后端 Agent 写代码前必须读取所属项目的 `AGENTS.md`，并按其中规范实现、测试和交接；不要随意引入新的 API 风格、客户端生成器、状态库或测试框架，除非项目已有明显方向或用户明确要求。

采用契约优先的对接方式：

- 后端变更必须同步更新接口文档。
- 前端必须消费文档中的接口契约，不要猜字段、路径或错误结构。
- 发现契约不一致时，记录问题、分派责任角色，并从源头修正。
- Agent 交接只报告改动文件、关键行为、验证结果和阻塞；长日志、完整源码和大段接口内容留在文件或工具输出中。

## 最终回复

总结时必须包含：

- 执行模式：单工作区并行或 Codex 桌面版跨线程协作。
- 使用的前端项目地址和后端项目地址。
- 生成或更新的接口文档位置。
- 后端完成内容和前端完成内容。
- 已运行的测试、自动化 smoke、浏览器 E2E 结果和集成检查。
- 剩余阻塞或需要用户处理的事项。
